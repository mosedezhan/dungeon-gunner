# 02 — 平衡评审走静态分析子代理

## 背景与动机

项目的平衡迭代在 `add-ai-workflow-foundation` 后已经形成一个轻量闭环：

```
想法 → 改 src/config.js → 浏览器手测 → /balance-tune（why + after-feel 必填） → 重复
```

`why` 和 `after-feel` 都是**定性**口语描述，没有任何数值锚点。这带来三个长期问题：

1. **横跳**：同一个参数被反复回调——上周觉得过强，这周觉得过弱；新会话回看 `balance-journal` 只看到几行口语化记录，没有客观依据复盘"上次为什么觉得过强"。
2. **耦合归因**：当一次手测同时影响多个维度（如 `damage` ↑ + `fireRateMs` ↓），after-feel 无法分摊到具体维度，调整链路变成黑箱。
3. **跨会话失锚**：一周后回看条目，"节奏松了"是相对当时还是相对绝对？玩家当时刚拿哪种升级路线？信息已蒸发。

balance-journal 的定性事实层是项目的纪律基线，**不能也不应**被替代——只有真人手测能验证体感。但定性事实需要一个量化对照层来止血上述三个问题。本 ADR 决定如何引入这个对照层。

## 决策与权衡

### 选型对比：A 静态分析 / B headless 模拟 / C 运行时埋点

会话中梳理过三条路：

**A. 纯静态分析子代理**

读 `src/config.js` 字段值，按公式算派生指标（玩家 DPS、敌人 EHP、TTK、技能 uptime 等），输出"调前 → 调后"指标对比。零运行时改动、零代码、零依赖。
- ✅ 实施成本极低（纯 markdown prompt）。
- ✅ 与项目"无构建步骤、无 node 依赖"约定相容。
- ❌ 抓不到走位/AI 涌现行为，pierce / multishot 在敌密度低场景下被高估。
- ❌ 不计算敌人组合权重；面对敌种 spike 的体感无法预测。

**B. Headless 模拟 + 评审子代理**

把 Player / Enemy / Bullet 的纯逻辑层从 Phaser 渲染层抽出，用脚本化策略（"原地射击"、"理想走位"等 baseline）跑 N 次每波，产出 TTK 分布、死亡率等。子代理拿模拟结果做评审。
- ✅ 能反映涌现行为（敌人组合、AI 走位、命中率折扣）。
- ❌ 工程量大：需重构出可独立运行的逻辑层；当前代码 `Player` / `Enemy` 与 `scene.physics` 强耦合，解耦本身就是一个 OpenSpec change。
- ❌ 维护成本高：每次新增机制（新敌种、新升级）都要同步模拟层。
- ❌ "理想走位"策略本身是另一个建模问题——错的策略给错的预测。

**C. 真人 telemetry**

在 `GameScene` 埋点把战斗事件写到 localStorage，每次手测后子代理读日志做事后分析。
- ✅ 数据真实，反映真实玩家手感。
- ❌ 改 `src/scenes/GameScene.js` 增加事件钩子 + localStorage 持久化 + 跨会话日志聚合，是一条独立工程线。
- ❌ 依赖玩家真去打、且打到代表性场景；样本稀疏时分析价值低。
- ❌ 与 balance-journal 的"调一次留一痕"短闭环不匹配——埋点数据要积累几次才有意义。

### 决策：首版选 A，刻意不为 B/C 预留接口

- **A 的边际成本几乎为零**——只是一个子代理 prompt + 一个斜杠命令，没有运行时改动。即便后续证明价值不足，删除成本也是零（删三个 markdown 文件）。
- **B 和 C 都需要先动游戏代码**，violate 本变更 "零代码影响" 的设计目标。把它们留作独立 change 候选（`add-balance-review-simulation` / `add-balance-telemetry`）。
- 三方案不是替代关系而是补充关系：A 给"调前预测 + 调后对照"，B 给"批量场景下的统计真相"，C 给"真人样本下的实测真相"。先把 A 做扎实，等使用中暴露明确的局部正确-全局错判时再评估升级。
- **不为 B/C 预留接口**：YAGNI。如果未来要做 B，是另一个独立 change，重新走 propose → design → apply 流程。本 change 的子代理 prompt 不需要兼容"未来可能传模拟数据进来"——那种兼容是过度设计。

### 配套决策

- **子代理 + 斜杠命令双层结构**：子代理隔离上下文，避免污染主会话；斜杠命令做参数解析与 diff 拉取。两层都是 markdown，零代码。
- **双模式触发（diff / what-if）**：diff 模式服务"调完看预测"，what-if 服务"调前看决策依据"。两者覆盖典型工作流。
- **与 `/balance-tune` 解耦**：不强制集成。集成会破坏 balance-tune 现有 UX 与必填字段约束；解耦保留两者各自的纪律。如未来证明价值高，单独立 change 整合。
- **不输出具体数值建议**：子代理只给定性方向（"风险偏强"），不给"应该调到 X.X"。量化结果不能掩盖手测体感的不可替代性，否则会让 after-feel 退化为"对照预测打勾"。
- **disclaimer 强约束**：子代理输出末尾固定附"本评审为静态推算，不替代 playtest 体感"。这层心理安全垫即便公式扩展后也保留。

## 适用边界

**适用：**

- 单参数或多参数的 `src/config.js` 数值调整。
- 假设性 what-if（"如果调到 X 会怎样"）。
- 结合 `/balance-tune` 的工作流——可前可后，对照定性 after-feel 做"预测 vs 实测"复盘。
- 教学/复盘场景：回看历史 balance-journal 时跑 what-if 重建当时的指标快照。

**不适用：**

- **结构性平衡变化**：加新字段、改公式、加维度。这种走 OpenSpec change（参考 `RULES.md` 颗粒度策略）。子代理遇到白名单外字段会输出 `WARNING` 并跳过，不会静默给错。
- **`UPGRADES` 数组的改动**：UPGRADES 是 `apply` 函数语义而非数值，本子代理不分析。改动 UPGRADES 应走 OpenSpec change。
- **跨文件平衡问题**：如"敌人 AI 在某种地形下表现异常"的体感问题——这是逻辑层问题，不在 config 范围。本子代理不会反推代码逻辑。
- **绝对值判断**："X 是不是合理值"——子代理只给相对变化（Δ）的解读，不给绝对值的好坏判断。绝对值判断仍是手测的事。
- **多敌种组合涌现效应**：子代理对每敌种独立算 TTK，不算"波次平均 TTK"——均值会掩盖体感 spike。

## 后续观察项

- **使用率**：`/balance-review` 在前 5 次调参中被实际调用的频次。若 < 50%，说明集成度不够（解耦的代价显化），考虑下一个 change `add-balance-review-into-tune` 把它接进 `/balance-tune` 的流程。
- **预测 vs 实测的偏差模式**：在 5-10 次"调前评审 + 调后 after-feel"配对中，偏差是否系统性偏向某方向？若指标恒高估玩家能力（pierce / multishot 的近似偏差），是否需要给公式加"敌密度折扣"参数。
- **新字段的扩展频率**：项目下次加 `armor` / `critRate` / `shieldHp` 这类新字段时，**必须同步更新 `.claude/agents/balance-review.md` 的字段白名单与公式**——否则子代理会静默输出 WARNING，但用户可能没注意到。本 ADR 在此显式留 hook：未来加这类字段的 OpenSpec change 必须把"更新 balance-review 白名单"作为一项任务。
- **是否需要升级到 B/C**：若发现 A 给出的"局部正确"经常引导出"全局错判"（典型场景：敌人组合下的体感与单敌种 TTK 严重不一致），那是触发 `add-balance-review-simulation` 的信号。先用 5-10 次实际使用积累证据再决，避免基于直觉提前升级。
- **子代理 model 选择**：起步用 `sonnet`。若使用频率高且公式集稳定后，可下调到 `haiku` 节省成本——评审是结构化任务，haiku 应该够。下调单独走一个 micro-change（不必新立 OpenSpec change，直接改 frontmatter 即可，但要在 `balance-journal` 留一条元注解条目）。
- **disclaimer 是否产生疲劳**：用户重复看几十次后是否开始无视？若是，下一次更新可考虑把 disclaimer 收敛为一行短句（仍保留语义但减少噪音）。
