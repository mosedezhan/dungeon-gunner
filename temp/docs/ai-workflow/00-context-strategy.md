# 00 — 上下文策略与变更颗粒度

> ADR（决策叙事）。硬规则见 [`RULES.md`](./RULES.md)。

## 背景与动机

本项目（dungeon-gunner，Phaser 3 小游戏）有双重目标：
1. 产出可玩的 Roguelike 弹幕游戏
2. 探索 AI 辅助开发的最佳实践（项目即教材）

预期会经历多轮（潜在 20+ 次）会话迭代。多轮迭代的真正瓶颈不是单次对话的上下文窗口，而是**跨会话的知识断层**——每次会话重新解释历史决策的成本会逐步压垮干活的时间。

具体痛点：
- 调参高频（一次会话可能 5-10 次微调），但每次靠记忆判断"为什么调"
- 复杂子系统的局部规则（场景流转、对象池）若全塞进根 CLAUDE.md 会让其膨胀
- 框架引入决策若无显式记录，三个月后无法回答"我们当时为什么这么搭"
- OpenSpec 已就位但 specs/changes 都是空的，没建立纪律即开工等于裸奔

## 决策与权衡

### 决策 1：变更分流，不是所有改动都走 OpenSpec

**选择**：建立颗粒度策略表，把改动分成 6 类，各走不同流程。
**拒绝**：
- "一切走 OpenSpec"——纯数值微调走完整流程，流程感大于产出感
- "一切凭直觉"——失去归档、可追溯、上下文压缩的好处
**关键权衡**：纯数值平衡走 `balance-journal.md` 而不是 change，是给迭代频率让路；但**结构性变化**（改公式、加维度）必须走 OpenSpec，避免暗中扩大变化。

### 决策 2：archive 即时性 + 进行中 ≤ 2 个

**选择**：变更完成立即归档，`changes/` 进行中数量上限 = 2。
**拒绝**：
- "想归就归"——纪律会松懈，已完成 change 滞留就是上下文负债
- "≤ 1 个"——某些情况下需要并行（如工作流 change + 游戏 change），太严
**关键权衡**：archive 是 OpenSpec 的杀手特性（被动上下文压缩），充分利用它需要纪律。设硬数字让规则可执行可检查。

### 决策 3：CLAUDE.md 分层加载，根 CLAUDE.md 保持轻量

**选择**：根 CLAUDE.md 只放全局架构 + 工作流指针；子系统规则下沉到 `src/<subdir>/CLAUDE.md`，按需加载。
**拒绝**：
- "全部塞进根"——多轮迭代后必膨胀
- "每个子目录都加"——过度设计
**关键权衡**：起步只在 `scenes/` 和 `entities/` 加分层（最复杂的两个子系统），新增分层必须留 ADR。

### 决策 4：规则与决策叙事分离

**选择**：硬规则放 `RULES.md`（给 AI 读），决策叙事放 `NN-*.md`（给人读）。
**拒绝**（这是 apply 阶段识别的修正）：
- "规则和叙事都塞进 NN-*.md 四节"——文件性质混乱：是给 AI 读的政策，还是给人读的历史档案？
- "规则全部进根 CLAUDE.md"——和游戏架构耦合，CLAUDE.md 变成混合体
**关键权衡**：根 CLAUDE.md 只放 4 行指针告诉 AI 去读 `RULES.md`，工作流内容仍有 enforcement 路径，CLAUDE.md 保持游戏导向。

### 决策 5：拒绝 BMAD / Spec-Kit / 多代理框架

**选择**：单人小项目用 OpenSpec 已够。
**拒绝**：
- BMAD 多角色（Analyst/PM/Architect/Dev）——单人开发不需要角色扮演
- GitHub Spec-Kit——和 OpenSpec 哲学重叠，叠加会撕裂心智
- CrewAI / AutoGen——程序化多代理过重，本期无场景
**关键权衡**：等真正主对话装不下大改动时再考虑 Claude Code 子代理（subagents），不是 BMAD。

### 决策 6：拆 3 个相邻 change，不一次做完

**选择**：`add-ai-workflow-foundation` → `establish-baseline-specs` → `add-debug-overlay` 三个连续 change。
**拒绝**：
- "一个大 change 包含全部"——schema 错位（工作流 vs 游戏能力混在一份 specs 里）
- "完全不走 OpenSpec，直接写文件"——丢失"项目即教材"的递归示例价值
**关键权衡**：每个 change 自洽满足 ≤ 3 天 + 独立 playtest，archive 后语义清晰。

## 适用边界

本决策约束的是**项目自身的迭代流程**，不约束：
- 第三方依赖的版本管理（Phaser 3 升级走单独 change）
- 临时探索性脚本（不进 src/，不留痕）
- 用户文档 / README 内容更新（本身就是"留痕"）

颗粒度的硬规则（≤ 3 天、≤ 5 specs、≤ 2 进行中）是纪律不是法律。极少数情况可破例（重大重构），但破例必须在该 change 的 `design.md` 显式说明理由。

## 后续观察项

- `/onboard` 命令读取 archive "最近 N 个"中 N=3 是否合适？
- 进行中 ≤ 2 个 这条硬规则在并行做"工作流 change"和"游戏 change"时会不会卡？
- balance-journal 增长到上百条后是否需要归档/分文件？
- 决策记录数量 10+ 时是否需要在 `RULES.md` 或本目录加索引？
- 决策 4 引入了 `RULES.md`，运行一段时间后看 AI 是否真的会主动去读它（如果不读，需要更强的 enforcement，比如改 `/onboard` 强制加载）。
