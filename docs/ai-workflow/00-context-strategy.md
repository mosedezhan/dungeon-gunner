# 00 — 上下文策略与变更颗粒度

> ADR（决策叙事）。硬规则见 [`RULES.md`](./RULES.md)。

## 背景与动机

游戏开发项目预期经历多轮 AI 辅助迭代。核心瓶颈不是单次对话的上下文窗口，而是**跨会话的知识断层**——每次重新解释历史决策的成本逐步压垮干活时间。

## 决策与权衡

### 决策 1：变更分流，不是所有改动都走 OpenSpec

建立颗粒度策略表，把改动分成 6 类，各走不同流程。结构性变化（改公式、加维度）必须走 OpenSpec。

### 决策 2：archive 即时性 + 进行中 ≤ 2 个

变更完成立即归档，`changes/` 进行中数量上限 = 2。设硬数字让规则可执行。

### 决策 3：CLAUDE.md 分层加载

根 CLAUDE.md 只放全局架构 + 工作流指针；子系统规则下沉到 `src/<subdir>/CLAUDE.md`。

### 决策 4：规则与决策叙事分离

硬规则放 `RULES.md`（给 AI 读），决策叙事放 `NN-*.md`（给人读）。

### 决策 5：轻量 OpenSpec 优先

用 OpenSpec 的 spec-driven schema 管理变更，不引入更重的框架。

## 适用边界

约束项目自身的迭代流程。不约束第三方依赖管理、临时探索脚本、用户文档更新。
